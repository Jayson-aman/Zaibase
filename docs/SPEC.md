## 0. これは何か（Why）

建設業界の **資材不足・人手不足・余剰資材の廃棄** を解決するプラットフォーム。
ナフサ・石油不足による資材難への対応も目的。職人（個人事業主）同士の **互助** を軸にする。

- アプリ名: **Zaibase（ザイベース）**
- キャッチコピー: 「Connect. Build. Share.」
- **ミッション（改訂版）:** 「日本の建設技術を、次の世代へ渡す。国境を越えて。日本人職人が誇りを持って働ける国を守る。そして共に働く外国人の夢も、一緒に叶える。Zaibaseはすべての職人のそばにいる。」
  - 正当な報酬が必ず手元に届く（エスクロー・未払い防止）
  - 技術が次の世代に受け継がれる（師弟マッチング・技術継承記録・国境を越えた技術輸出）
  - 一人親方が安心して独立できる（労災・社保・確定申告サポート）
  - 「建設業はかっこいい」を発信する（誇り投稿・若者向けコーナー）
  - **誰も一人にしない** — 日本人職人も、ベトナム・インドネシアなど共に働く外国人も

## 4. 新機能（本セッションで追加・要件定義）

### 4-1. 生活ライン保証マッチング（労力シェア）
職人が「30日のうち最低何日働けば生活できるか（生活ライン）」を登録。
- 生活ラインに**未達の人へ優先的に仕事を回す**
- 仕事が余る人は**余剰を不足する人へシェア**
- 全員が生活ラインを下回らないようにする互助設計
- 優先対象: 出戻り（仕事が飛んだ人）・無職状態の人・一人親方

**設計メモ（要決定）**
- 仕事の出どころ: 職人同士の応援 / 元請の仕事 / 両方 →（ヒアリング中）
- 稼働日数のカウント: 自己申告 / アプリ自動 / 両方 →（ヒアリング中）
- 生活ラインの公開範囲: 非公開 / 状態のみ / 公開 →（ヒアリング中）

### 4-2. スポット応援マッチング（空き時間 × 移動時間）
「15:30から空き」のように空き時間を登録 → 近くの応援募集現場へ。
- **3時間前に告知メール**を自動送信
- **時給換算**で参加
- **Google Maps で移動所要時間を算出** → 実働可能時間と想定報酬を自動計算
  - 例: 15:30空き／現場18:00まで／移動35分 → 16:05着 → 約1h55m稼働 → 時給×時間 で報酬目安
- **遠い現場は表示しない**（距離フィルター）

**設計メモ（要決定）**
- 通える範囲: 30km / 50km / 80km / 車で◯分 / 職人ごとに設定 →（ヒアリング中）
- ⚠️ コンプラ: 「時給で時間管理」は雇用・派遣に寄るリスク。**時間単価の業務委託（指揮命令なし）** として設計し専門家確認。

### 4-3. 役所手続きサポート（システム化）
職人の **社会保険・税務署届出** を Zaibase で支援。
- 対象: 開業届 / インボイス登録番号 / 労災（特別加入）/ 社会保険 / 確定申告
- ⚠️ 現実: 「役所へ全自動送信」は不可。税理士法・社労士法で代行は有資格者のみ。
- 取れる道（3段階）:
  1. **書類作成＋手続きガイド＋期限リマインド**（すぐ可能）
  2. e-Tax / マイナポータル / 会計ソフト（freee・マネーフォワード）API連携
  3. 提携税理士・社労士への依頼窓口（合法的に代行）
- ①で作った書類が「登録ゲート」の証明にもなる。

### 4-4. 多言語対応
対応言語: 🇯🇵日本語 / 🇻🇳ベトナム語 / 🇰🇭カンボジア語（クメール語）/ 🇮🇩インドネシア語 / 🇬🇧英語
- **画面UIの文字**: 言語ごとの翻訳ファイル（i18n）で即時切替
- **ユーザー投稿の自由文（出品・チャット）**: **Vertex AI（翻訳/Gemini）で自動翻訳**
- 画面上部に 🌐 言語切替

## 6. 手数料体系（改訂案 v3 — 二層構造・2026年6月）

> **労務（スポット応援）** と **請負工事（専門工事）** で手数料を分離する。請負は単価が桁違いに大きいため**逓減方式**。旧案（一律16〜25%）・v2一律8%は廃止。  
> アプリ内シミュレータ：マイページ「💰 手数料シミュレータ」

### 6.1 設計思想（なぜ二層＋逓減か）

| 比較 | 労務マッチング | 請負工事 |
|---|---|---|
| 典型単価 | ¥8万/件 | ¥300万/件 |
| 手数料率 | 10%（職安法上限10.8%内） | **4%前後（逓減）** |
| 1件収益 | ¥8,000 | **¥120,000**（15倍） |

**逓減にする理由**

1. **発注者離脱防止** — 5,000万×8%＝400万は現実的でない  
2. **大型案件の誘引** — 「大きいほどお得」で元請・大手の登録を促す  
3. **不当利益リスク回避** — 工事代金に対する高率徴収の問題を避ける  

**役割分担**

- **労務** ＝ ユーザーを集める入口（件数・認知）。単価は低いが職安法枠内  
- **請負** ＝ 収益の柱。件数は少なくてよい  

### 6.2 フェーズ別課金

| フェーズ | 手数料 |
|---|---|
| **β（現在）** | **0%**（許可・弁護士確認まで凍結） |
| **許可取得後** | 下表（労務・請負それぞれ届出・契約上の整理が必要） |

### 6.3 労務マッチング手数料（スポット・小額請負）

有料職業紹介に該当する場合の枠。**上限10.8%**。

| 項目 | 値 |
|---|---|
| 標準率 | **10%** |
| プロ会員 | **9%**（−1%） |
| マックス会員 | **8%**（−2%） |
| 最低手数料 | **¥500/件** |
| 法定上限 | 報酬 × **10.8%** |

### 6.4 請負工事手数料（逓減・専門工事直接請負）

**法的整理（要弁護士確認）**：「有料職業紹介」ではなく**工事請負契約の仲介手数料**として設計。現行モデルC（専門工事分離・直接請負）が前提。

| 請負金額（税抜想定） | 手数料率 |
|---|---|
| 〜¥100万 | 8% |
| ¥100万〜¥500万 | 5% |
| ¥500万〜¥2,000万 | **4%** |
| ¥2,000万〜 | **2.5%** |

**チーム・組合受注（§19）**：契約当事者が労働者協同組合の場合も上表の逓減率を適用。例：¥2,000万×2.5%＝**¥50万/件**（個人¥100万案件の8%＝¥8万と比較して収益インパクト大）。

| 会員割引 | 全ティアから −0.5%（Pro）/ −1.0%（Max） |
|---|---|
| 最低手数料 | ¥3,000/件 |

**例**：¥300万請負 → 4% ＝ **¥120,000/件**

## 7. サブスクリプション（改訂案 v4）

**2軸モデル**：① **受注者（職人）向け** Pro/Max（本章） ② **発注者（元請・現場）向け**（§7.5・別料金でARPU向上）

### 7.0 解約されにくい設計の核心

最強の解約防止は **「やめると損になる構造」**。

| 軸 | 仕組み | Zaibaseでの実現 |
|---|---|---|
| **手数料割引の自走** | 払うほど儲かる | Maxの請負−1%は単価が大きいほど効く。300万×3件/月＝**月9万円節約**、月額6,980円差し引きで**約+8.3万円**の合理的継続 |
| **データ蓄積ロック** | 記録が資産になる | 案件履歴・取引先・評価（第2期：GPS出退勤・CCUS実績）が積み上がるほど、解約＝「自分の仕事の記録が見えなくなる」心理的コスト＋競合移行コスト |

サブスク単体（試算約34万/月）は請負手数料（約120万/月）の補助輪。**本質価値は手数料ROI＋マッチング成功率＋蓄積データ**。

### 7.2 フリープラン（ベース）の設計思想

**体験版ではなく「不便な実用版」**。「無料で十分」ではなく **「無料だと損」** を体感させる。

| 制約 | 内容 | 意図 |
|---|---|---|
| **応募制限** | 月**5件**まで（案件への応募・問い合わせ） | 使えるが本格利用には足りない |
| **手数料+2%** | 労務・請負とも標準率に+2%上乗せ | 取引が増えるほど有料化のメリットが明確 |
| 労務の上限 | +2%後も **職安法10.8%上限**でキャップ | 法適合。請負側の+2%が主な損失体感 |

例：請負300万の場合、ベース6%（4%+2%）＝18万 vs Max 3%（4%−1%）＝9万 → **1件で9万円差**。

投稿（案件登録）はベースでも可。制限は **応募側** に置く（受注者のフリープラン設計）。

### 7.4 解約防止設計

#### A. 解約時に失うもの

| 資産 | ダウングレード時 |
|---|---|
| 手数料優遇 | +2%（ベース）に戻る → 請負1件で数万〜十万の差 |
| バッジ・優先枠 | 信頼シグナル・問い合わせ減 |
| 応募無制限 | 月5件に制限 |
| 実績・履歴の可視化 | 3件表示・1年制限 |
| 蓄積データ（第2期） | GPS/CCUSは閲覧・エクスポート不可 |

## 15. 法務コンプライアンス（最優先3点）

> 弁護士（建設業・労働法）レビュー前の設計メモ。正本はガイドライン第27〜30条・利用規約第9〜11条。

### ① 人材マッチングの法的スキーム（職業安定法・労働基準法）

**背景**：労働基準法第6条により「業として他人の就業に介入して利益を得る」行為は、法律で許される場合を除き禁止。職業紹介事業は職業安定法上の許可が必要。**申請窓口は本社所在地の都道府県労働局**（神奈川本社なら神奈川労働局）だが、許可自体は**全国で事業可能な国の許可**。

**Zaibaseにとっての最重要論点**：**有料職業紹介では港湾運送業務・建設業務は取り扱えない**（職業安定法上の制限）。建設現場の職人マッチングを「有料職業紹介」でカバーすることはできない。許可申請より先に、採用スキームの確定が必要。

| スキーム | 内容 | 許可 | Zaibaseでの位置づけ |
|---|---|---|---|
| **A. 募集情報等提供** | 案件情報の掲載のみ | 不要 | 手数料が取りにくい。情報プラットフォームとしての下限 |
| **B. 業務委託仲介** | 労働契約に該当しない請負・業務委託の仲介 | **不要の可能性**（職業安定法非適用） | **最有力**。一人親方同士の業務委託契約仲介として設計。労働契約性を有する案件の厳格排除が必須 |
| **C. 有料職業紹介** | 他人の就業に介入し手数料を得る | 厚生労働大臣許可（審査2〜3ヶ月） | **建設業務には使えない**。雇用紹介を扱う場合のみ検討 |

- 無許可で職業紹介に該当する収益化は第32条違反リスク（1年以下懲役または100万円以下罰金）
- 許可要件（スキームCの場合）：基準資産500万・現預金150万、職業紹介責任者、労働局申請
- **現状**：手数料課金凍結。βは無料マッチングのみ。UI・契約はスキームB（請負・業務委託のみ・雇用は扱わない）で設計済み

| モデル | 必要な許可・スキーム |
|---|---|
| 雇用（正社員・アルバイト）を仲介 | 有料職業紹介許可（建設業務は対象外） |
| 一人親方の請負・業務委託マッチング | **スキームB**（要弁護士確認。職業安定法の潜脱とみなされない実態設計が条件） |
| 一時雇用して現場に送る | 労働者派遣許可（別途・禁止設計） |

### ④ 二軌制：個人マッチング × 労働者協同組合（§19）

Zaibaseは**2つの法的スキームを並行**する設計とする。混同しないことが最重要。

| 軌道 | 契約当事者 | 法的根拠 | 典型案件規模 |
|---|---|---|---|
| **軌道1：個人** | 一人親方 ↔ 発注者 | スキームB（業務委託仲介）§15① | ¥30万〜¥100万 |
| **軌道2：組合** | 労働者協同組合 ↔ 発注者 → 組合内で従事分量配当 | 労働者協同組合法（2022）§19 | ¥500万〜¥2,000万 |

- 軌道1は**今すぐ**弁護士確認の対象。軌道2は**組合機能の画面・課金実装前**に弁護士＋社労士確認が必須。
- Zaibaseの役割：どちらの軌道でも**マッチング・工程・証跡・決済インフラ**を提供。組合の設立・登記・内規策定の代行はしない（有資格者連携）。

## 19. 労働者協同組合スキーム（法的正本）

> **チーム結成・発注UIは実装済み**（Firestore + Functions）。**本番課金・組合名義契約**は弁護士ゲート通過後。正本は本節。

### 19.1 労働者協同組合とは（労働者協同組合法・2022年10月1日施行）

**基本原理**：組合員が**出資**し、それぞれの意見を反映して組合の事業が行われ、**組合員自らが事業に従事する**組織。

一言で言うと「**出資・経営・労働を全員で担う法人**」。株式会社と違い、出資が多い人が偉いわけではない。

| 要点 | 内容 |
|---|---|
| 議決権 | 出資額にかかわらず**1人1票**（議決権・選挙権平等） |
| 出資 | 法律の定めなし。1人1口¥1万〜¥5万の例あり |
| 性質 | **非営利法人**。出資額に応じた配当は**禁止** |
| 分配 | **従事分量配当**は可能（働いた分に応じた配当） |
| 設立 | **発起人3人以上**。準則主義（行政許認可不要）。要件を満たし**登記**で法人格付与 |
| 資本金 | 株式会社のような最低資本金制限なし |

**建設業との接点**：組合員を「労働者」として組合に従事させることで**雇用保険・社会保険の適用**が可能。一人親方が「会社に所属しているように見える」偽装一人親方問題を、**実態ある組合**として正面から回避しうる。

### 19.2 法人格の選択肢（Zaibaseが想定する組合形態）

| 選択肢 | 法律 | 組合員雇用 | Zaibase |
|---|---|---|---|
| **① 労働者協同組合** | 労働者協同組合法 | ○ 想定どおり | **本スキームの正本** |
| ② 事業協同組合 | 中小企業等協同組合法 | × 想定外 | 共同購買・保険加入の補助のみ |
| ③ 株式会社・合同会社 | 会社法 | ○ | コスト重い。Phase 3以降の代替 |

### 19.3 現状と将来の構造

```
【現状：個人バラバラ（軌道1）】
一人親方A → 小案件しか取れない
一人親方B → 閑散期がある
一人親方C → 大手から相手にされない

【将来：Zaibase × 労働者協同組合（軌道2）】
一人親方A・B・C・D・E
  → Zaibase上で「タイル職人組合 横浜」を結成（＝組合設立・登記は組合側の手続き）
  → 大型マンション外壁工事 ¥2,000万 を組合が受注
  → Zaibase：工程管理・報酬分配計算・CCUS・保険手続きガイド・エスクロー
  → 従事分量配当で各組合員に分配
```

| 関係 | 当事者 |

| 対内（分配） | 組合 → 組合員（従事分量配当・給与/報酬） |

### 19.6 法的境界・リスク（設計で必ず守ること）

| 論点 | 整理 |
|---|---|
| Zaibaseの立場 | マッチング・証跡・決済の**プラットフォーム**。組合の設立・登記・内規・労務管理の**主体ではない** |
| 偽装組合 | 名前だけの組合・実態は指揮命令 → 偽装請負・偽装一人親方。組合員の**経営参加・議決**の実態が必要 |

| 資金フロー | エスクロー→組合口座→メンバー分配は**資金移動業・決済代行**の要否を弁護士確認（§16エスクローと同様） |

| 「Zaibaseチーム」 | アプリ上のチーム結成 ≠ 法人格。法人化（登記）完了まで**軌道2の正式契約は組合名義不可** |

## 20. 多層下請・透明性管理（建設業法第22条）

## 26. 在庫・物流方針（倉庫を持たない）

> **結論：課金前・軌道前は自前倉庫を持たない。** ZaibaseはITプラットフォームのまま、問屋の物流インフラを借りる。

### 26.1 倉庫を持つと何が変わるか

| 項目 | 目安コスト |
|---|---|
| 倉庫賃料 | 月¥30〜200万 |
| 入出庫人件費 | 月¥30〜50万 |
| 在庫リスク（売れ残り） | 全損の可能性 |
| 設備（フォークリフト・棚・WMS） | 初期¥500万〜 |
| 損害保険 | 月数万円 |
| **合計固定費** | **最低 月¥100万〜** |

課金未開始の段階で負うと、**物流会社化**し固定費で潰れる。Amazon・楽天も最初は在庫を持たないモデルから開始した。

### 26.2 在庫ゼロの3方式（優先順）

| 優先 | 方式 | 内容 | フェーズ |
|---|---|---|---|
| **①** | **ドロップシッピング** | 職人注文 → Zaibaseが問屋へ自動発注 → 問屋が直送。在庫・倉庫ゼロ | **今すぐ** |
| ② | 問屋倉庫の在庫引当 | 「月〇〇個分をZaibase用に確保」契約。保管は問屋、出荷も問屋 | 月間取引¥500万超 |
| ③ | 3PL委託 | アスクル型に倉庫・配送・在庫管理を委託 | 月間取引¥1,000万超 |

**初手は①のみ。** 山本政清商店・ミウラ等との契約で即開始可能（§25・`docs/wholesaler-call-yamamoto-miura.md`）。

### 26.3 自前倉庫を検討するライン

```
今（無料β）     → ドロップシッピングのみ
月取引¥500万超  → 問屋への在庫引当契約
月取引¥3,000万超 → 3PL委託を検討
月取引¥1億超    → 初めて自前倉庫を検討
```

### 26.4 グループ購買の配送（倉庫なしで解決）

| 方式 | 説明 |
|---|---|
| A 個別配送 | 各自の自宅・事務所へ。送料は個別、倉庫不要 |
| B 現場一括 | 指定日・現場へ一括搬入。`#deliveryScreen`・電子伝票と連携 |
| C 店頭受取 | ミウラ・山本政清等の店頭受取（地域限定・送料ゼロ・当日可） |

「10人でまとめ買い・配送先バラバラ」は **A or B or C** の組み合わせで対応。中間倉庫は不要。

### 26.5 実装との対応

- 資材EC：**モデルB（ドロップシッピング）＋モデルC（グループ購買）** のみ本番化（§22.1）
- モデルA（在庫型仕入れ）：**月取引¥1億超まで凍結**
- 注文フロー：`materialsOrders` → 問屋API/メール発注 → 問屋直送（在庫フィールドなし）
- UI文言：資材マーケット・問屋交渉画面に「在庫ゼロ・ドロップシッピング」を明示

## 33. 子育て互助システム（#childcareScreen）

> 建設業の慢性的な人手不足の根本原因のひとつが、**ひとり親・父子家庭の就業困難**。  
> 保育所・幼稚園の送り迎えが現場の早朝開始・延長と両立できず、離職・参入断念が起きている。  
> Zaibaseユーザー同士の**互助（互いに助け合う）**で、育児と仕事の両立を支援し、業界全体の稼働率を上げる。

### 33.1 社会課題と設計思想

| 課題 | 対策 |
|---|---|
| 朝7時現場集合 × 保育所8時受付 → 無理 | 近くのZaibaseユーザーに送りを頼む |
| 急な工期変更 → 夕方迎えに行けない | 緊急アラートで即時援助者を募集 |
| 互助してもメリットがない | 援助ごとにZaibaseポイント+150ptを付与 |
| 相手が信頼できるか不安 | Zaibase本人確認済みユーザーのみ参加可 |
| 法的リスク | 互助の範囲（有償旅客運送に該当しない設計）・弁護士確認推奨 |

**ミッションとのリンク**：「誰も一人にしない」「一人親方が安心して独立できる」

### 33.2 機能一覧

| 機能 | 概要 |

### 33.5 法的注意事項（⚠️ 弁護士確認推奨）

- **道路運送法の有償旅客運送との関係**：ユーザー間の互助報酬は「友人・知人間の実費弁償的な費用負担」の域を想定。定期的・反復的な有償送迎は「自家用有償旅客運送」や「ライドシェア」の規制対象になる可能性があるため、弁護士確認を行う。

## 32. 環境法令コンプライアンス（水質汚濁防止法・土壌汚染対策法）

> 解体・土工・地盤改良・コンクリート工事では建設業法・労働安全衛生法に加え、**環境二法**への対応が必要。  
> 実装正本：`functions/env_compliance.js`・`#complianceGateScreen`（環境法令セクション）

### 32.1 対象法令と適用場面

| 法律 | 主要条文 | 対象工種 | Zaibaseでの対応 |
|---|---|---|---|
| **水質汚濁防止法**（昭和45年法律第138号） | 第5条（特定施設届出）・第12条（排水基準） | コンクリート洗浄・掘削汚濁水排出・プラント設置 | 届出確認書を `verificationDocs.waterQuality` に記録・期限管理 |
| **土壌汚染対策法**（平成14年法律第53号） | 第3条（特定施設廃止調査）・第4条（形質変更届出）・第16条（汚染土搬出届） | 解体・土工・地盤改良・掘削・残土搬出 | 調査報告書を `verificationDocs.soilContamination` に記録・汚染土出品完全ブロック |

### 32.2 水質汚濁防止法（第5条・第12条）

**概要**：工場・事業場が公共水域または下水道に汚水・廃液を排出する場合、指定施設（特定施設）の設置・変更・廃止は都道府県知事への**届出が義務**。

建設現場における該当ケース：
- コンクリートプラント（生コン製造設備）
- コンクリートミキサー・打設機器の洗浄水を公共水域へ排出
- 掘削汚濁水（ベントナイト液・泥水）の処理・排出
- 基礎工事・地盤改良における薬液注入後の排水

**届出が不要なケース**：閉鎖系処理（タンク収集・専門業者回収）で公共水域に一切排出しない場合。

### 32.3 土壌汚染対策法（第3条・第4条・第16条）

**第3条（特定施設廃止時の調査義務）**：有害物質を製造・使用した特定施設を廃止（または土地を売却）する際、都道府県知事に土壌汚染状況調査報告書を提出。
- **建設との接点**：工場解体・旧事業所の解体工事前に、土地所有者が特定施設を有していたか確認が必要。

**第4条（形質変更時の届出）**：3,000㎡以上の土地の形質を変更（掘削・造成）する場合、着工30日前までに届出。
- **建設との接点**：大規模造成・大型基礎工事・地盤改良工事。

**第16条（汚染土壌の搬出届出）**：汚染が確認された土壌を搬出する場合は事前届出必須。

**Zaibaseの汚染土ブロック**：残土・購入土の出品フローで土壌調査報告書または非汚染証明のない案件を完全ブロック（既存実装・資材出品ガイドライン §15）。

### 32.4 コンプライアンス実装（登録ゲート連携）

| 機能 | 実装場所 | 動作 |
|---|---|---|
| 届出書類の自己申告登録 | `#complianceGateScreen` → 環境法令セクション | `verificationDocs.waterQuality` / `soilContamination` を Firestore 保存 |

```

  質問2: 睡眠が十分に取れているか
```

### 34.4 退職金・経済設計シミュレーション

```
入力: 現在の月収・建退共掛金月額・現在年齢・退職予定年齢
計算:
  建退共推計額 = 掛金月額 × 12 × 積立年数 × 利率補正
  年金収入（概算） = 月収 × 0.005 × 加入年数
  iDeCo推計額 = 月額 × 12 × 年数 × 1.03^年数（複利3%仮定）

```

## §35 CSR・ステークホルダー管理（`#csrScreen`）

### 35.1 概要

建設業の社会的責任（CSR）をISO 26000の7原則に基づき可視化・管理する画面。
説明責任・透明性・倫理的行動・ステークホルダー尊重・コンプライアンス・国際規範尊重・人権尊重の7カテゴリで自己評価を行い、スコア化する。

### 35.2 タブ構成

| タブ | 内容 |
|---|---|
| 📋 CSRチェック | ISO 26000対応チェックリスト（7カテゴリ）・スコアリング |
| 🤝 ステークホルダー | 利害関係者（発注者・下請・一人親方・従業員・地域等）別対応状況 |
| 👤 人権DD | 政府ガイドライン（2022年）準拠・5ステップ人権デューデリジェンス |
| ⚥ ジェンダー | 女性活躍推進法・育児介護休業法・男女均等法等チェック |
| 🔒 フリーランス法 | フリーランス保護法（2024年11月1日施行）義務・推奨事項チェック |

### 35.3 CSRスコアリング

CSR_CATEGORIES（7カテゴリ・20項目）の各チェック項目に配点（4〜6pt）を設定し、
`取得点数 / 満点 × 100` でスコアを算出する。

| スコア | 評価 | 表示色 |
|---|---|---|
| 70%以上 | 優良 | 青（#2563eb） |
| 40〜69% | 要改善 | 黄（#f59e0b） |
| 39%以下 | 要対応 | 赤（#ef4444） |

### 35.4 人権DD 5ステップ（政府ガイドライン2022年準拠）

1. 人権方針の策定・公表
2. 人権リスクアセスメント
3. リスク軽減措置の実施
4. モニタリング
5. 情報開示・コミュニケーション

各ステップに「実施済み」チェックおよびメモフィールド（3〜4項目）を備える。

### 35.5 フリーランス保護法チェック

特定受託事業者に係る取引の適正化等に関する法律（2024年11月1日施行）の
義務項目（第3条〜第8条）および推奨項目を網羅。

義務項目未対応の場合は警告ボックスを表示。

### 35.6 Cloud Functions（`functions/csr.js`）

| 関数名 | 概要 |
|---|---|
| `saveCsrCheck` | CSRチェック結果（checkState・genderState・flState・score）を `csrChecks/{uid}` に保存 |
| `getCsrReport` | `csrChecks/{uid}` と `humanRightsDd/{uid}` を結合して最新CSRレポートを返す |
| `saveHumanRightsDd` | 人権DD 5ステップデータ（ddState）を `humanRightsDd/{uid}` に保存 |

## §36 工数管理（#koushouScreen）

### 概要
建設産業1,945h問題（全産業平均1,680hより265時間多い）への対応。2024年4月より建設業にも時間外労働上限規制（年360h/特別条項720h）が適用された。案件別工数入力・見積 vs 実績比較・上限アラートを提供する。

### 4タブ構成
- **日次入力**: 日付・案件名・時間・作業種別・メモを入力してFirestoreに保存
- **案件別**: 案件ごとの累計工数と目標工数の比較・進捗バー表示
- **月次レポート**: 選択月の日別・作業種別内訳グラフ
- **上限管理**: 年間時間外累計と法定上限のゲージ・警告表示

### 上限規制の基準値
- 年間時間外上限（通常）: 360h
- 年間時間外上限（特別条項）: 720h
- 月間時間外上限（通常）: 45h
- 月間時間外上限（特別条項）: 100h
- 建設業への適用開始: 2024年4月

### Firestoreスキーマ
**koushouEntries** コレクション:
- uid: string（ユーザーID）
- date: string（YYYY-MM-DD）
- jobName: string（案件名、最大80文字）
- hours: number（0.5単位）
- workType: string（作業種別）
- memo: string（メモ、最大200文字）
- createdAt: Timestamp

**koushouTargets** コレクション（ドキュメントID = uid）:
- [jobName]: number（案件ごとの目標工数）

### Cloud Functions
- **addKoushouEntry**: 工数エントリをバリデーション後にFirestoreへ保存
- **getKoushouReport**: 指定年の月別集計・年間時間外合計を返す
